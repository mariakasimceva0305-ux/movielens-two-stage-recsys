# MovieLens Two-Stage Recommender System

## Кратко
Двухэтапная рекомендательная система на MovieLens: быстрый этап отбора кандидатов, затем более точное переранжирование ограниченного набора объектов и выдача рекомендаций через API.

## Задача
Полное попарное ранжирование всех объектов даёт высокую вычислительную стоимость. В этом проекте используется двухэтапная схема: сначала отбирается ограниченный набор кандидатов, затем для него применяется более точная модель переранжирования. Такой подход позволяет улучшать качество верхних позиций списка рекомендаций без полного перебора всех объектов.

## Что улучшено
- вычислительная нагрузка снижена за счёт быстрого предварительного отбора кандидатов;
- качество верхних позиций повышается за счёт отдельного этапа переранжирования;
- качество оценивается по набору метрик top-K, а не по одной метрике.

## Архитектура
```text
[история взаимодействий]
          |
          v
[подготовка признаков]
          |
          v
[быстрый отбор кандидатов]
          |
          v
[точное переранжирование]
          |
          v
[список рекомендаций top-K]
          |
          v
[API выдачи рекомендаций]
```

## Структура репозитория
```text
app/        API-сервис
artifacts/  сохранённые артефакты
src/        обучение, признаки, оценка, ранжирование
data/       данные
notebooks/  исследовательские материалы, если используются
```

## Критерии качества
- Precision@K
- Recall@K
- MAP@K
- NDCG@K
- задержка ответа

## Ключевой вывод
Двухэтапная схема позволяет удерживать вычислительные затраты под контролем и одновременно улучшать качество верхних позиций рекомендаций. Основной вклад в качество даёт точный второй этап, а первый этап отвечает за то, чтобы не потерять релевантные кандидаты слишком рано.

## Запуск
```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

## Ограничения
- качество второго этапа ограничено качеством первого: если релевантный объект не попал в кандидаты, переранжирование уже не поможет;
- итоговая конфигурация зависит от размера каталога и требований по задержке;
- для производственной системы потребуются отдельные механизмы обновления признаков и мониторинга качества.
